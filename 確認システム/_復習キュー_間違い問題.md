後で「復習ノート作って」と言えば、これを元に苦手復習ノート（×△のみ・○は載せない）にまとめる。

## 103 企業経営理論

### 2026-06-14 採用・教育訓練（○13 △0 ×1）
- **× OJTの長所として、個々の能力・進度に合わせた指導ができる点が挙げられる。**
  - 正答：**正**（これは正しい記述）。OJTの長所＝マンツーマンで**個別ペースに対応・実務直結**。

### 2026-06-14 労働基準法・労災（○13 △0 ×1）
- **× 労災保険の保険料は、労働者と事業主が折半して負担する。**
  - 正答：**誤**。労災保険料は**全額事業主負担**（労働者負担はゼロ）。
  - 比較：雇用保険は労使双方負担／健康保険・厚生年金は労使折半。**労災だけ全額事業主**と覚える。

## 106 経営情報システム

### 2026-06-14 データ分析基盤（○22 △5 ×1）※弱点はDWMの性質に集中
- **× DWHのデータは、一度格納されると基本的に更新されず、時系列で積み上げられていく（非揮発性）。**
  - 正答：**正**。DWHの4特性＝**サブジェクト指向・統合・時系列(時変)・非揮発（更新しない）**。
  - 「貯めて分析する箱」なので過去データを書き換えない＝非揮発。これが時系列分析の土台。
- **△ DWHは、業務系データを時系列で蓄積し意思決定支援のために分析する目的で構築される。**
  - 正答：**正**（DWHの定義そのもの）。目的＝**意思決定支援（分析）**。
- **△ DWHは、更新・削除を頻繁に繰り返す日常業務のトランザクション処理に最適化された仕組みである。**
  - 正答：**誤**。それは**OLTP**。DWHは更新せず蓄積し分析する**OLAP系**。**DWH≠OLTP**を取り違えない。
- **△ スノーフレークスキーマは、ディメンションテーブルを正規化して階層的に分割した構造である。**
  - 正答：**正**。スター＝非正規化で高速／スノーフレーク＝正規化で冗長減・結合増。
- **△ ビッグデータの「3V」とは、Volume(量)・Variety(多様性)・Velocity(速度)である。**
  - 正答：**正**。量・種類・発生速度。＋Veracity(正確性)・Value(価値)で5Vとも。
- **△ データレイクに生データを溜め込むだけで管理を怠ると「データスワンプ（沼）」化を招く恐れがある。**
  - 正答：**正**。メタデータ管理・カタログ化を怠ると「何があるか分からない沼」に。

### 2026-06-14 IoT・先端技術（○17 △10 ×1）※用語の自信不足・通信/構成要素が弱い
- **× ブロックチェーンは、中央集権的な管理者が単一の台帳をすべて一元管理する仕組みである。**
  - 正答：**誤**。中央管理者を置かず**P2Pで分散管理（分散台帳）**。だから耐改ざん・高可用。
- **△ センサーは物理量（温度・光等）を電気信号に変換して取得する装置である。**
  - 正答：**正**。センサ＝現実をデータ化する「目・耳」（入力側）。
- **△ アクチュエータは、温度や光などの物理量を計測してデータ化する装置である。**
  - 正答：**誤**。それは**センサー**。アクチュエータは信号→物理動作（モーター・バルブ）の「手足」（出力側）。**センサ⇔アクチュエータの向きを逆にしない**。
- **△ LPWAは、低消費電力で広範囲をカバーする反面、通信速度は低速である。**
  - 正答：**正**。電池長持ち・広域・**低速小容量**（センサーデータ向け）。
- **△ LPWAは大容量・高速通信に最適化され、高精細動画配信に向く。**
  - 正答：**誤**。LPWAは低速・小容量。動画など大容量は不向き。
- **△ 5Gの主な特徴は、高速大容量・低遅延・多数同時接続である。**
  - 正答：**正**。この**3点セット**で暗記。
- **△ 5Gは4Gより同時接続できる端末数が少なくなった。**
  - 正答：**誤**。5Gは**多数同時接続**が強み（IoT多数機器を収容）。
- **△ ブロックチェーンは前ブロックのハッシュを含むため改ざんが極めて困難。**
  - 正答：**正**。1つ改ざんすると以降全ハッシュが崩れ検知できる。
- **△ スマートコントラクトは契約条件をプログラム化しブロックチェーン上で自動執行する。**
  - 正答：**正**。条件成立で仲介者なしに自動実行。
- **△ NB-IoTやLoRaWANは、LPWAの代表的な通信規格である。**
  - 正答：**正**。いずれも低電力・広域のIoT向け規格。

### 2026-06-14 Web技術（○11 △15 ×2）※基本概念が浅い・要復習
- **× HTTPはステートレスで各リクエストは独立し前後の状態を保持しない。**
  - 正答：**正**。だからCookie/セッションで状態を補う。「ステートレス＝状態を持たない」を押さえる。
- **× JSONは人にも機械にも読みやすい軽量なデータ記述形式でWeb APIでよく使われる。**
  - 正答：**正**。key:value（波括弧）の軽量形式。
- **△ HTTPステータスコードの200番台はサーバ内部エラーを示す → 誤**
  - 正答：200番台＝**成功**。サーバ内部エラーは**500番台**。404＝Not Found。
- **△ サーバサイドスクリプト(PHP等)はブラウザ上で実行され結果をサーバへ送る → 誤**
  - 正答：サーバサイドは**Webサーバ側で実行**しHTMLをブラウザへ返す。ブラウザ実行はクライアントサイド(JS)。
- **△ XMLは固定タグのみで独自タグは定義できない → 誤**
  - 正答：XMLは**独自タグを定義できる拡張可能(eXtensible)**なマークアップ言語。
- **△ HTTPは常に暗号化するのでHTTPのままでもパスワードを安全に送れる → 誤**
  - 正答：HTTPは**平文**。暗号化は**HTTPS(SSL/TLS)**。
- 上記以外の△（正答=正）：HTTPS/セッション/GET・POST/404/3層/REST/Ajax/クライアントサイド/SSL証明書/アクセシビリティ/サードパーティCookie ＝定義の再確認を。

### 2026-06-14 プロジェクト管理（○15 △13 ×0）※EVM計算・PERT/短縮技法が穴
- **△ クリティカルパスとは最も余裕(フロート)が多い経路 → 誤**
  - 正答：**所要時間が最長＝余裕ゼロ**の経路がCP。ここが全体の最短完了を決める。
- **△ LOC法は設計初期でも正確に算出できるのが長所 → 誤**
  - 正答：初期は行数不明で**精度が低い**。言語依存もある。機能量で測るのがFP法。
- **△ EVMのコスト差異CVは AC−PV で求める → 誤**
  - 正答：**CV＝EV−AC**（出来高−実コスト）。プラスなら予算内。式を取り違えない。
- **△ クラッシングとは作業を一部並行して短縮する技法 → 誤**
  - 正答：それは**ファストトラッキング**。**クラッシング＝資源(人員)を追加投入**して短縮。両者を区別。
- **△ CPIが1を下回ると予算に余裕がある → 誤**
  - 正答：**CPI<1は予算超過気味**（効率が悪い）。1超で予算内。
- **△ RADは時間をかけ大規模に一括開発する手法 → 誤**
  - 正答：RADは**短期間・迅速開発**を重視（プロトタイプ/ツール活用）。
- 上記以外の△（正答=正）：FP法/EVM定義/SV(EV<PVで遅れ)/ファストトラッキング/3大制約/最早=最遅はCP上/スコープクリープ ＝定義の再確認を。

### 2026-06-14 記憶装置（○8 △20 ×0）※論点全体が未定着＝実質未学習。要・本格ノート＋再テスト
誤答(×)ゼロだが△20で確信ゼロ。特に引っかけで揺れた急所：
- **△ SRAMはリフレッシュ必要・低速・安価で主記憶に使う → 誤**
  - 正答：**SRAMはリフレッシュ不要・高速・高価でキャッシュ向き**。リフレッシュ必要・安価・主記憶はDRAM。両者を逆にしない。
- **△ EEPROMは紫外線でのみ消去できるROM → 誤**
  - 正答：**EEPROMは電気的に消去・書換可能**。紫外線で消去するのはEPROM。
- **△ SSDは機械駆動部が多くHDDより消費電力・発熱が大きい → 誤**
  - 正答：**SSDは駆動部なし・低消費電力・静音・耐衝撃**。
- **△ 光ディスク(CD/DVD/BD)は磁気で読み書きする → 誤**
  - 正答：**レーザー光**で読み書き。磁気はHDD・磁気テープ。
- **△ ライトバックは書込みのたび両方へ即時書込み → 誤**
  - 正答：それは**ライトスルー**。ライトバックはキャッシュにのみ書き後でまとめて主記憶へ反映。
- 上記以外の△（正答=正）：記憶階層/キャッシュの役割/DRAM=主記憶/SRAM=キャッシュ/揮発性/フラッシュ/HDDアクセス時間(シーク+回転待ち)/参照の局所性/ライトスルー/インタリーブ/RAID0・1・5 ＝**論点まるごと要復習**。

### 2026-06-14 データベース設計（○13 △15 ×0）※正規化目的・関係演算が穴
- **△ 候補キーのうち主キーに選ばれなかったものを外部キーと呼ぶ → 誤**
  - 正答：それは**代替キー**。外部キーは**他表の主キーを参照する列**で別概念。
- **△ 正規化を進めるほど表は減り必ず検索が速くなる → 誤**
  - 正答：正規化で**表は増え**、結合が増えてむしろ遅くなることも（あえて非正規化する場合あり）。
- **△ 射影とは条件に合致する行を抽出する操作 → 誤**
  - 正答：それは**選択(selection)**。**射影(projection)＝列（属性）を取り出す**。選択＝行・射影＝列。
- **△ インデックスを多く張れば検索も更新も全て高速でデメリット無し → 誤**
  - 正答：更新時の索引メンテ・記憶領域増という**デメリットあり**。読みは速いが書きは重くなる。
- **△ 正規化の主目的は検索速度の最大化そのもの → 誤**
  - 正答：主目的は**重複排除と更新時異常の防止（データ整合性）**。速度向上が主目的ではない。
- 上記以外の△（正答=正）：参照整合性/正規化の定義/第1〜第3正規形/選択・射影の正しい定義/インデックスの効果/排他制御/スキーマ ＝定義の再確認を。
