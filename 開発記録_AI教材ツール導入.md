## 1. 目的（当初ゴール）

VS Code ＋ AIツール ＋ API を使って、**中小企業診断士の暗記HTML教材**を作る。

- **コスト最小の使い分け戦略**
  - 基本70%：Gemini Flash（無料枠）— テキスト整理・基本HTML
  - 高品質20%：Gemini Pro — 複雑なJS・深い文脈読み取り
  - 最終手段＆画像10%：OpenAI GPT / DALL·E 3 — 画像生成・デバッグ
- **教材の中身**
  - スマホ対応・モダン・絵文字
  - 暗記用イラストの埋め込み（DALL·E等で生成。「サイヤ人とゴミ箱の公園」は一例）
  - クリックで答えが出る「ミラー穴埋め（アコーディオン）」
- **Live Server でホットリロード**しながら開発、エラーはAIに自己修正させる

> ※「Cline（Roo Code）を使うこと」自体が目的ではなく、上記の教材を上記コスト戦略で作ることが本来の目的。ツールは手段。

## 4. つまずきと原因・解決（重要な学び）

### (A) 間違ったキーを貼っていた
- `.env` の `GEMINI_API_KEY` に、誤って **GitHubトークン（`ghp_`始まり）** を貼っていた。
- Geminiキーは **`AIza` または `AQ.` 始まり**。正しいキーに直して解決。
- 教訓：キーの**先頭文字**で種類を確認する。

### (B) Roo Code が 0% のまま固まる（本命の原因）
- 実際のエラー：**`Could not find ripgrep binary`**（開発者ツールのConsoleで判明）
- 原因：新しいVS Codeは ripgrep の同梱場所が **`@vscode/ripgrep-universal`** に改名。
  Roo Code 3.54.0 は**旧名 `@vscode/ripgrep` を探す**ため見つけられず、タスクが始められない。
- キー・ネットワーク・ストリーミングは **curlで全て正常**と確認済み（＝Roo Code固有の問題）。
- 解決を試みたが：
  - Roo Code に更新版なし（既に最新）
  - `/Applications` のVS Code本体はmacOS保護（App管理）で**書き換え不可**（symlink作戦は「Operation not permitted」）
  - 本体コピー改変はVS Code更新のたび再対応が必要で**非推奨**
- → **別系統の Continue に乗り換えて回避**（こちらは正常動作）。

### (C) セキュリティ対応
- `.gitignore` に `.env` の記載が無かった → **`.env` `.env.*` を追記**（OpenAIキー等の流出防止）。
- `.env` は元々Git未追跡＝**GitHubには漏れていない**ことを確認済み。
- スクショに `.env` のキー断片が写ったため、**キーの作り直し（ローテーション）を実施予定**（再開時の最初のタスク）。

## 6. ツール比較メモ（Roo Code vs Continue）

| 観点 | Roo Code | Continue |
|---|---|---|
| 種類 | Cline派生・自律エージェント | OSSのAIコード支援（老舗） |
| 導入数（目安） | 約1.7M | 約3.3M |
| 自律性 | 高い（丸投げ向き） | 中（対話・確認しながら） |
| コード補完 | ほぼ無し | あり |
| キー保存 | VS Code暗号化（安全） | config.yamlに平文 |
| 暴走リスク | 高め（自動承認時） | 低め |
| 今回のripgrep問題 | 発生 | 起きにくい |
| 初心者の入りやすさ | △ | ○ |

- 認知度・安定実績 → **Continue**
- キー保管の堅さ → **Roo Code**
- 暴走しにくさ（初心者の安全）→ **Continue**
- 結論：**まず1つ作るなら Continue で十分**。量産で全自動化したくなれば Roo Code を直して再挑戦。

## 8. 再開時の手順（次にやること）

### ステップ A：キーを安全に作り直す（最初にやる）
1. https://aistudio.google.com/apikey で**新しいGeminiキー**を作成・コピー
2. `.env` の `GEMINI_API_KEY=` を新キーに差し替え→保存→タブを閉じる
3. （担当AIが）新キーをcurlでテスト＋ `~/.continue/config.yaml` を新キーへ同期
4. Continueで動作確認できたら、AI Studioで**古いGeminiキーを削除**
5. （任意）OpenAIキーも同様に作り直し（platform.openai.com）

### ステップ B：教材づくり本番
1. 元ノートのテキストを Continue に読み込ませる（`@ファイル名`）
2. 「アコーディオン式の暗記HTMLにして」と指示（スマホ対応・絵文字・モダン）
3. Live Server（Go Live）でプレビューしながら調整
4. 暗記イラストは OpenAI/DALL·E で生成するスクリプトを作って実行 → `<img>`で埋め込み
5. エラー時：モデルをPro/GPTに切替（⌘'）＋エラーを貼って自己修正

## 9. 市場ポジション・差別化メモ（将来のサービス化の種）

### 着眼点
「AIと一緒に、語呂イメージ画像つきの“自分専用インタラクティブHTML暗記ノート”を、自分の手で作り上げる」というワークフロー全体を、**商品／コーチングとしてパッケージ化したもの**は、まだ世に薄い。

### 市場の現状（部品は存在するが、組み合わせは未成熟）
- 暗記SaaS：Anki / Quizlet / Monoxer(モノグサ＝暗記のサービス化で普及)
- AIでノート→学習教材：Google NotebookLM / Quizlet AI / ChatGPT等（急増中）
- トグルで答えを隠すノート：Notion / Obsidian / Scrapbox
- 診断士e-learning：スタディング等
- → 「暗記」「AI生成」「クリックで答え」は個別には存在。だが
  「**自作・自己ホスト × 語呂/キャラ画像のAI生成 × 作り方を教えるコーチング**」の交差点は薄い。

### なぜ「まだ無い」か
1. 参入障壁が高い（VS Code/APIキー/拡張機能＝今回ハマった所）。だから製品はノーコードSaaSに向かう。
2. トレンドが逆方向（製品は「全部隠して簡単に」、本件は「作れる力をつける」＝思想が逆）。
3. 自作=売る物が無く収益化しにくいので、誰も商品化を急がない。

### 差別化の核（言語化）
- 売りは「個別最適(personalization)」ではない（SaaSも適応学習で寄せてくる）。
- **価値は「成果物」でなく「作る過程」そのもの。自分で作る＝最強の勉強**（教育学の「生成効果 generation effect」）。
- SaaSが便利になるほど、逆に「自分で作れる人」の理解の深さが際立つ。

### AGI時代の位置づけ（むしろ追い風）
- 資格は規制・信頼・役割の装置（士業）→ AIが解けても人間の資格要件は残る。
- AIを使う側にも理解・判断・説明責任が残る。
- 学ぶこと自体に意味（規律・達成・アイデンティティ）。
- → 暗記の価値は下がるが「**自分で学びを設計し、AIと共に作るメタ能力**」の価値は上がる。
  コーチングは「教材の作り方」を超え「**AI時代の自走する学び方を教える**」ものになる。

### リスクと対策
1. 規模はニッチ（多数は楽なSaaSへ）→ コーチングはニッチで十分成立（少人数高単価）。
2. 「なぜ自作？」に一文で答える＝“作る過程が学び”。
3. 陳腐化対策：売るのは特定ツールでなく「考え方・型・つまずき回避」（今日もRoo Code→Continueと変えた）。

### 最大の資産
今回ユーザー自身が通った「初心者が必ずハマる所（キー取り違え・ripgrep問題・ツール選定）」の実体験こそ、コーチングの一番の素材。
