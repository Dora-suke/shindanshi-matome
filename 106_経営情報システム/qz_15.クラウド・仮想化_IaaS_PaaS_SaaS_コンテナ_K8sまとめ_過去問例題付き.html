📝 過去問エッセンス(H21-17) (エッセンス)

Q. クラウドコンピューティングの特徴として最も適切なものはどれか。

  • ア. ネットワーク経由でITリソースをオンデマンドかつ従量課金で利用できる
  • イ. 利用者が自社でサーバを購入・設置し全てを管理する形態である
  • ウ. クラウドでは複数のテナントがリソースを独占的に使用する
  • エ. 利用量に関わらず定額料金のみで提供されるのがクラウドの特徴である
👁 答えを見る
✅ 正解:ア  クラウドのNIST定義の核心は「オンデマンドのセルフサービス・従量課金・リソース共用・弾力性・幅広いNWアクセス」の5特性。
  • イ ✗ ✗:正しくはそれはオンプレミス（自社運用）の説明でクラウドとは逆の形態
  • ウ ✗ ✗:正しくはクラウドはマルチテナントでリソースを複数ユーザーが共用する方式
  • エ ✗ ✗:正しくはクラウドは使用量に応じた従量課金が基本で定額固定ではない

📝 過去問エッセンス(H22-11) (エッセンス)

Q. サーバ仮想化の説明として最も適切なものはどれか。

  • ア. 複数の物理サーバを1台の強力なサーバに物理的に統合する技術
  • イ. 1台の物理サーバ上でハイパーバイザを使い複数のOSを同時に稼働させる技術
  • ウ. コンテナと同義であり、軽量な仮想環境をOSカーネルを共有して実現する
  • エ. ネットワーク機器をソフトウェアで制御するSDNと同一の概念
👁 答えを見る
✅ 正解:イ  サーバ仮想化はハイパーバイザが物理資源を管理し、その上に複数のVM（仮想マシン）を稼働させる技術。各VMは独立したOSを持つ。
  • ア ✗ ✗:正しくはそれは物理的なサーバ統合であり仮想化（ソフトウェアによる分割）とは異なる
  • ウ ✗ ✗:正しくはコンテナはOSカーネルを共有するが仮想マシン（VM）は独立したOSを持つ別概念
  • エ ✗ ✗:正しくはSDNはネットワーク制御の概念でサーバ仮想化とは別の技術領域

📝 過去問エッセンス(R03-3) (エッセンス)

Q. コンテナ技術の特徴として最も適切なものはどれか。

  • ア. コンテナは仮想マシン（VM）と同様に独自のOSカーネルを持つ
  • イ. コンテナは起動が遅く数分かかるためリアルタイム処理には不向き
  • ウ. ホストOSのカーネルを共有するため軽量で高速に起動できる
  • エ. Dockerはコンテナのオーケストレーションツールであり複数ホストを管理する
👁 答えを見る
✅ 正解:ウ  コンテナはホストOSカーネルを共有するためゲストOS不要で軽量・高速（秒単位の起動）。一方VMは独立したOSを持ち隔離度が高いが重い。
  • ア ✗ ✗:正しくはOSカーネルを持つのはVMでコンテナはホストOSカーネルを共有する
  • イ ✗ ✗:正しくはコンテナの特長の一つが秒単位の高速起動でありVMより圧倒的に速い
  • エ ✗ ✗:正しくはDockerはコンテナのビルド・実行ツールであり複数ホスト管理はK8sが担う

中身の荷物（アプリ）は箱から出さないので、引っ越し先（別のサーバ）でもすぐ動く。これが「移植性が高い」の正体。

👩‍🏫
先生からひとことVMは「家ごと引っ越す」だから重いし時間がかかる。コンテナは「箱だけ運ぶ」から軽くて速い。ただし箱なので、家ほど頑丈には隔てられない（＝分離度はVMより低い）と覚えよう。

🖥️ 仮想化って結局なに？

1台の物理サーバを「分割」して、複数の仮想マシン（VM = Virtual Machine）として使う技術。リソースを無駄なく使える。

  • VM ＝ OSごと丸ごとの仮想コンピュータ
  • コンテナ ＝ OS（カーネル）は共有して、アプリだけ箱に詰める軽量な方式
VM=家ごと／コンテナ=部屋だけ分離の単位

重い／起動分単位
分離度：高

コンテナ

軽い／起動秒単位
分離度：VMより低

  • ア. SaaSではOSのパッチ適用は利用者が行う
  • イ. PaaSではインフラだけでなくアプリケーションも事業者が管理する
  • ウ. IaaSではOSもアプリもすべて事業者が管理する
  • エ. PaaSではOSやミドルウェアを事業者が管理し、利用者はアプリ開発に集中できる

  • ア ✗ ✗:正しくはSaaSではOSを含むインフラ全てを事業者が管理し利用者はデータ設定のみ担う

📝 過去問エッセンス(H27-13) (エッセンス)

Q. ハイパーバイザ型仮想化（ホスト型）の説明として最も適切なものはどれか。

  • ア. ホストOSの上にハイパーバイザを置き、その上に仮想マシンを稼働させる方式
  • イ. 物理ハードウェア上に直接ハイパーバイザを置き高性能を実現するベアメタル型
  • ウ. OSカーネルを共有し個別のOSを持たない軽量なコンテナ方式
  • エ. ネットワーク経由で遠隔サーバを仮想的に接続するVPN方式
👁 答えを見る
✅ 正解:ア  ホスト型仮想化はWindows等のホストOS上にVMwareやVirtualBoxを入れ、その上でゲストOSを動かす。ベアメタル型よりオーバーヘッドが大きい。
  • イ ✗ ✗:正しくはそれはベアメタル型（ネイティブ型）でハイパーバイザがハードに直接乗る方式
  • ウ ✗ ✗:正しくはそれはコンテナ方式の説明でOSカーネルを共有する全く別の技術
  • エ ✗ ✗:正しくはVPNは暗号化されたネットワーク接続の技術で仮想化方式とは無関係

📝 過去問エッセンス(R02-13) (エッセンス)

Q. VMとコンテナの比較として最も適切なものはどれか。

  • ア. コンテナはVMより高い隔離度（セキュリティ分離）を持つ
  • イ. VMは独立したOSを持つため隔離度は高いが起動時間はコンテナより長い
  • ウ. コンテナは独自のカーネルを持つためVMと同等のリソース消費がある
  • エ. VMは起動が高速でコンテナより効率よくスケールアウトできる
👁 答えを見る
✅ 正解:イ  VMはゲストOSを持つため完全に隔離されるが数分かかる起動が欠点。コンテナは軽量・高速だが同一カーネルを共有するため隔離度はVMより低い。
  • ア ✗ ✗:正しくはコンテナはOSカーネルを共有するためVMより隔離度が低くセキュリティ分離は弱い
  • ウ ✗ ✗:正しくはコンテナは独自カーネルを持たずホストOSカーネル共有のため軽量である
  • エ ✗ ✗:正しくはVMは起動が遅く（分単位）コンテナの方が秒単位で高速にスケールできる

📦 VM vs コンテナ（最重要対比）

項目仮想マシン（VM）コンテナ
OSゲストOSを各VMに持つホストOSのカーネルを共有
サイズGB単位（重）MB単位（軽）
起動時間分単位秒単位
分離度強い（OSごと隔離）VMより弱い
移植性低め高い（イメージ持ち運び）
軽さ・速さ・移植はコンテナ／分離はVM使い分け

📝 過去問エッセンス(R03-7) (エッセンス)

Q. KubernetesとDockerの役割分担として最も適切なものはどれか。

  • ア. DockerはコンテナのオーケストレーションツールでKubernetesはコンテナの作成ツール
  • イ. KubernetesはDockerの上位互換であり両方を同時に使う必要はない
  • ウ. DockerはコンテナのビルドとパッケージングをしKubernetesは複数コンテナを管理・調整する
  • エ. どちらも仮想マシンを管理するためのツールであり用途は同じ
👁 答えを見る
✅ 正解:ウ  Docker＝コンテナを作る・動かすツール。Kubernetes（K8s）＝複数コンテナを多数のホストに自動配置・スケール・自己修復するオーケストレーター。
  • ア ✗ ✗:正しくはDockerがビルド・実行でK8sがオーケストレーションの役割で逆の説明
  • イ ✗ ✗:正しくはDockerとK8sは補完関係にあり現実には組み合わせて使われる
  • エ ✗ ✗:正しくはDockerもK8sもコンテナ技術に特化したツールで仮想マシン管理とは異なる

🐳 Docker と Kubernetes の役割

  • Docker：コンテナを作る・動かすためのツール。イメージからコンテナを起動
  • Kubernetes（K8s）：多数のコンテナを自動配置・自動復旧・スケールするオーケストレーション基盤
  • K8sの基本単位は Pod（1つ以上のコンテナをまとめた最小単位）
Docker＝作る／K8s＝束ねる役割分担

🔧 コンテナのメリット・デメリット

✅ メリット

  • 軽量（OSを持たない）／即時起動（秒単位）
  • 高い移植性：イメージをそのまま別環境で実行可
  • DevOps親和性：CI/CD パイプラインに乗せやすい
  • 同一ホストに多数を集積でき集約率◎

⚠️ デメリット

  • 分離度がVMより低い（カーネル脆弱性の影響）
  • 原則ホストOSと同じカーネルのOSしか動かせない
  • 運用にはK8s等のオーケストレーションが必要

⚠️ 【1次】引っかけ：「コンテナはVMより高い分離度を持つ」は誤り。逆。

📜 宣言的設定（Declarative）とは

K8sは 「あるべき状態（desired state）」を宣言すると、システムが自動でその状態に収束させる仕組み。

  • 例：「Pod を3つ動かす」と宣言 → 1つ落ちても自動で1つ起動して3つに戻す
  • 対比：命令的（imperative）＝手順を逐一指示する従来型
「3つ動かせ」と書くだけ宣言的
📝 過去問エッセンス(R01-23) (エッセンス)

Q. サーバレス（FaaS）の説明として最も適切なものはどれか。

  • ア. サーバを自社に設置してOSやアプリを完全に自己管理する形態
  • イ. サーバレスではサーバが物理的に存在しない
  • ウ. サーバ管理を意識せず関数単位でコードを実行でき実行時間・回数で課金される
  • エ. FaaSはIaaSより下位レイヤーに位置するサービスモデルである
👁 答えを見る
✅ 正解:ウ  サーバレス（FaaS）は「サーバはあるが管理は事業者」の形態。AWS Lambda等が代表例。コード（関数）の実行時間・回数のみ課金。PaaSの延長上に位置する。
  • ア ✗ ✗:正しくはそれはオンプレミスの説明でサーバレスは管理をすべて事業者に委ねる
  • イ ✗ ✗:正しくはサーバは物理的に存在するが利用者がそれを意識・管理しなくてよいのがサーバレス
  • エ ✗ ✗:正しくはFaaSはPaaSの発展形でIaaSより上位レイヤーに位置するサービスモデル

🏗️ サーバレス／FaaS

サーバレス＝サーバ管理を意識せず関数単位でコードを実行する形態。FaaS（Function as a Service）とも呼ぶ。

  • 例：AWS Lambda／Google Cloud Functions／Azure Functions
  • 課金は実行時間・回数（待機中は0円）
  • 位置づけはPaaSの一形態（より上位）

📜 本文のカードをうめる：宣言的設定 ＆ サーバレス／FaaS

K8sはあるべき状態（desired state）を宣言すると、システムが自動でその状態に収束させる仕組み（＝宣言的設定／Declarative）。

  • 例：「Pod を3つ動かす」と宣言 → 1つ落ちても自動で1つ起動して3つに戻す（自己修復）
  • 対比：命令的（imperative）＝手順を逐一指示する従来型

サーバレス＝サーバ管理を意識せず関数単位でコードを実行する形態。FaaS（Function as a Service）とも呼ぶ。

  • 例：AWS Lambda／Google Cloud Functions／Azure Functions
  • 課金は実行時間・回数（待機中は0円）
  • 位置づけはPaaSの一形態（より上位）

データだけは常に利用者責任。SaaSでもデータは利用者が守る。インフラ構成をコードで記述・再現可能にする手法＝IaC（Infrastructure as Code／Terraform・Ansible等）

🎼 語呂で穴埋め

G1：クラウド3層
語呂「土・場・完」：土台＝IaaS／場＝PaaS／完成品＝SaaS。
G2：NIST 5特性
語呂「オン・幅・共・弾・計」：オンデマンド・幅広いNW・リソース共用・弾力性・計測されたサービス。
G3：配置モデルとK8s
語呂「公私＝ハイブリッド／複公＝マルチ」。
語呂「Pod・Sv・Dep・NS」：最小単位＝Pod／窓口＝Service／配置＝Deployment／分割＝Namespace。

✏️ 穴埋め 初級（用語確認）

G1：クラウド3層
サーバなどインフラを借りる＝IaaS。例：AWS EC2。
アプリ実行環境（OS＋ミドル）を借りる＝PaaS。例：Heroku。
完成アプリを使う＝SaaS。例：Gmail、Microsoft 365。
G2：仮想化方式
ハード直上で動きホストOS不要＝Type1（ベアメタル）。例：VMware ESXi。
ホストOS上で動く＝Type2（ホスト）型。例：VirtualBox。
カーネル共有で軽量＝コンテナ型。例：Docker。
G3：K8s基本概念
K8sの最小単位＝Pod。Pod群へのアクセス窓口＝Service。Podを管理し更新する単位＝Deployment。クラスタ内を論理分割するのはNamespace。

✏️ 穴埋め 中級（対比・引っかけ）

G1：責任分界
OSパッチを利用者が当てるのはIaaS。OS管理を事業者に任せられるのはPaaS以上。
いずれのモデルでもデータは利用者責任。
G2：VM対コンテナ
起動時間：VMは分単位／コンテナは秒単位。
分離度：VMは高い／コンテナはVMより低い。
サイズ：VMはGB級／コンテナはMB級。
G3：配置モデル
パブリックとプライベートを連携：ハイブリッドクラウド。
AWSとAzureを併用：マルチクラウド。
端末近くで処理＝エッジコンピューティング。クラウドより低遅延。

✏️ 穴埋め 上級（応用・複合）

G1：NIST 5特性
需要に応じてリソースを伸縮させるのは迅速な弾力性。利用量に応じた従量課金は計測されたサービス。複数利用者が物理資源を共有するのはマルチテナント（リソース共用）。
G2：Docker／K8s 応用
Dockerfileから作る不変テンプレ＝イメージ。イメージを公開・共有する場所＝レジストリ（例：Docker Hub）。
「Pod を3つ動かす」と書くだけで状態を維持する方式は宣言的設定。Podが落ちても自動的に補充される性質を自己修復と呼ぶ。
G3：サーバレス・IaC
関数単位でコードを実行する形態はサーバレス（FaaS）。AWSの代表サービスはLambda。
インフラ構成をコードで再現可能にする手法はIaC。代表ツール：Terraform。

💼 ミニ事例

Q1. ある中小ECが「サーバ調達期間ゼロ・繁忙期だけ増強」を狙う。最適は？

自社DC設置はリードタイム長く、繁忙期外は過剰。

A. パブリッククラウドのIaaS＋オートスケール。NIST 5特性のうち「迅速な弾力性」「計測されたサービス（従量課金）」の典型活用。
Q2. 機密データは社内、Web画面はクラウドにしたい。配置モデルは？
A. ハイブリッドクラウド。社内＝プライベート、Web＝パブリックを連携。
Q3. 開発チームが「Macで動いたのに本番Linuxで動かない」を解消したい。
A. Docker（コンテナ）でイメージ化して移植性を確保。「ローカルで動いたものがそのまま本番で動く」のがコンテナの利点。
Q4. 自動運転車のセンサ解析。クラウドに全送信すると遅延が問題。
A. エッジコンピューティングで車両側処理。低遅延・帯域節約。

事例エッセンス

機密は社内＋一般はクラウド＝ハイブリッドクラウド。
環境差分を解消＝コンテナ（移植性）。
低遅延要件＝エッジコンピューティング。
