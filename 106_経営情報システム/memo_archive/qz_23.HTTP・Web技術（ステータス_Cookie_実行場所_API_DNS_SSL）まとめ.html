📨 1-1. HTTPの性質（ここを外すと全部ズレる）

HTTP＝Webの「お願いと返事」のルール（プロトコル）

ブラウザが「このページちょうだい」とリクエストを送り、サーバが中身をレスポンスで返す。この一往復を支える約束事がHTTP。

性質意味だから何が起きる
ステートレス
（状態を持たない）		1回ごとに「はじめまして」。前のやり取りを覚えていない「誰がログイン中か」を保てない → Cookie/セッションで補う（次タブ）
平文（HTTP）中身がそのまま流れる＝盗み見できるパスワードは危険 → HTTPS（SSL/TLS）で暗号化
✕「HTTPは常に暗号化するのでパスワードを安全に送れる」＝暗号化はHTTPSの役目。素のHTTPは平文。
HTTPの一往復（ステートレス＝毎回まっさら）

📡 1-2. ステータスコード（番号帯で覚える）

先頭の数字＝「結果の大分類」。帯さえ覚えれば9割取れる

コード帯意味代表例
200番台成功200 OK
300番台リダイレクト（転送）301（恒久）/302（一時）
400番台クライアント側のエラー404 Not Found
500番台サーバ側のエラー500 Internal Server Error
💡 覚え方：4＝あなた（クライアント）のミス／5＝サーバ（向こう）のミス。404は「あなたが存在しないURLを叩いた」＝400番台。
✕「200番台はサーバ内部エラー」＝帯の取り違え。内部エラーは500番台。200は成功。
🔑 このタブの核
  • ステートレス＝HTTPは毎回まっさら。「覚える」仕組みは別途必要（→Cookie）
  • 4xx＝クライアントのミス／5xx＝サーバのミス。404は4xx（あなた）
  • HTTP＝平文／HTTPS＝SSL/TLSで暗号化。HTTP単体は安全ではない
📝 穴埋め：HTTPとステータス
HTTPは前のやり取りを覚えない＝ステートレス。状態はCookie/セッションで補う。
素のHTTPは平文、暗号化するのはHTTPS（SSL/TLS）。
成功＝200番台、リダイレクト＝300番台。
クライアント側エラー＝400番台（例：404 Not Found）。
サーバ側エラー＝500番台。
前タブの宿題を回収する。 HTTPはステートレス＝毎回「はじめまして」だった。これだと「さっきログインした人」が分からない。 そこで「あなたは○番のお客さんですね」と覚える仕組みが要る。それがCookieとセッション。旅でいうと、サーバが記憶を持つ段階。

🍪 2-1. Cookieとセッション（どこに何が残る？）

Cookie＝ブラウザ側の「番号札」、セッション＝サーバ側の「台帳」

Cookieセッション
発行サーバが発行サーバが管理
保存場所ブラウザ側に保存（小さなデータ）サーバ側に状態を保持
役割再訪したお客さんを識別する「番号札」複数リクエストにまたがり同一利用者の状態（ログイン等）を維持
💡 イメージ：受付で番号札（Cookie）をもらって持ち歩く。お店の台帳（セッション）には「3番のお客＝ログイン中」と書いてある。番号札を見せると台帳から本人が分かる。
✕「Cookieはサーバ側にのみ保存しブラウザには残さない」＝保存場所の逆。Cookieはブラウザ側。

🚫 2-2. サードパーティCookie（規制の文脈）

「今見ているサイト以外」が発行するCookie＝追跡用で規制強化中

ファーストパーティ

今いるサイト自身が発行。ログイン維持・カゴ保持など正規用途。

サードパーティ

別ドメイン（広告会社など）が発行。サイトをまたいだ行動追跡（広告）に使われ、プライバシー観点で規制・廃止の流れ。

🔑 このタブの核
  • Cookie＝ブラウザ側に残る番号札／セッション＝サーバ側の台帳
  • 2つはセット。ステートレスなHTTPに「記憶」を足す仕組み
  • サードパーティCookie＝別ドメイン発行・広告追跡で規制強化中
📝 穴埋め：状態管理（Cookie/セッション）
Cookieはサーバが発行しブラウザ側に保存される小データ（再訪識別）。
セッションは複数リクエストにまたがり同一利用者の状態をサーバ側で維持。
別ドメインが発行し広告追跡に使うのがサードパーティCookie（規制強化中）。
HTTPはステートレスなので、状態維持にはCookie/セッションが必要。
最重要ひっかけはここ。 サーバから返ってきたページを「誰が・どこで」処理して画面にするのか。 試験は「役割」と「実行場所（ブラウザ側か／サーバ側か）」をわざと入れ替えて×を作る。旅でいうと描画の段階。

🔌 4-1. API・REST（窓口）

REST API＝HTTPでリソースを操作する「設計様式」

API＝アプリ同士の窓口（注文口）。なかでもREST APIは、HTTPの仕組みに素直に乗せてリソース（データ）を操作するWeb APIの設計スタイル。

💡 RESTはプロトコルではなく設計様式（HTTPの上での「お作法」）。

📦 4-2. データ形式（運ぶ箱の形）

JSON＝軽い／XML＝拡張できる

JSON

軽量なkey:value形式。今の主流。読み書きが軽い。

vs

XML

独自タグを定義できる拡張可能なマークアップ（eXtensible）。

✕「XMLは固定タグしか使えず独自タグを定義できない」＝拡張可能性（eXtensible＝X）の否定。XMLはタグを自分で定義できる。

⚡ 4-3. Ajax（画面を止めずに裏で更新）

Ajax＝ページ全体を再読込せず、非同期で部分更新

普通の遷移 vs Ajax
全体リロード白くなって待つ
⇔
Ajax裏で取得→一部だけ差替え
💡 検索候補がリロードなしで出る／地図がぬるぬる動く＝Ajax的な部分更新。
🔑 このタブの核
  • REST API＝HTTPでリソース操作するWeb APIの設計様式（プロトコルではない）
  • JSON＝軽量key:value／XML＝独自タグ定義できる拡張可能なマークアップ
  • Ajax＝全体再読込せず非同期で部分更新

📕 5-1. DNSとURL（住所まわり）

DNS＝「名前→IPアドレス」の電話帳

用語役割
DNSドメイン名 → IPアドレスに変換（正引き）。人間用の名前を機械用の番号に直す
URLリソースの所在（住所）を表す
💡 正引き＝「名前→IP」（例：example.com→192.0.2.1）。逆引きは「IP→名前」。試験は正引きが基本。

🔒 5-2. SSL/TLSサーバ証明書（なりすまし対策）

証明書＝「このサイトは本物です」の公的な身分証

SSL/TLSサーバ証明書は、通信の暗号化に加えてサイトの正当性を証明し、なりすまし（偽サイト）対策になる。鍵マークの正体。

💡 役割は2つ：①通信を暗号化／②運営者が本物だと証明（なりすまし対策）。

🏗️ 5-3. 3層アーキテクチャ＋表側の整え方

システムを役割で3つに分ける

プレゼンテーション層画面・見た目
→
アプリケーション層処理・ロジック
→
データ層DB・保存
用語ひとことで
SEO検索結果で上位化する施策
CMS専門知識なしでサイトを更新できる仕組み

アクセシビリティ誰でも（高齢者・障害者含め）使えるようにする配慮
🔑 このタブの核
  • DNS＝ドメイン名→IP（正引き）／URL＝所在
  • SSL/TLS証明書＝暗号化＋正当性証明（なりすまし対策）
  • 3層＝プレゼン／アプリ／データ。SEO・CMS・レスポンシブ・アクセシビリティは「表側の整え方」
📝 穴埋め：DNS・SSL・構成
DNS＝ドメイン名→IPアドレス（正引き）／URL＝リソースの所在。

🃏 反転で覚える（クリックで答え）

404はクライアント／サーバ どちらのエラー帯？
クライアント側（400番台）存在しないURLを叩いた＝あなた側のミス。サーバ側は500番台。
Cookieはどこに残る？セッションは？
Cookie＝ブラウザ側／セッション＝サーバ側。番号札はお客が持ち、台帳は店が持つ。
PHP等のサーバサイドはどこで実行？
Webサーバで実行しHTMLを返す。ブラウザで動くのはJavaScript。
XMLは独自タグを定義できる？
できる（eXtensible＝拡張可能）。「固定タグだけ」は誤り。JSONは軽量なkey:value。
🔑 失点しないための3原則
  • 場所を疑え：ブラウザ側↔サーバ側のすり替え（Cookie/セッション/JS/PHP）
  • 役割を疑え：構造(HTML)／見た目(CSS)／動き(JS)の入れ替え
  • 帯を疑え：4xx＝クライアント／5xx＝サーバ。暗号化はHTTPSの役目

クライアントJSはブラウザ、サーバサイドはサーバで実行。
HTMLは構造、暗号化はHTTPSの役目。
