🔌 2-1. RESTとは何か

REST（REpresentational State Transfer）

WebAPIの設計思想・アーキテクチャスタイル。プロトコルではない点が試験頻出の引っかかり。

❌ 非RESTな設計

/getUserById?id=123 /createNewUser /deleteUserAccount
URLに動詞が入る
VS

✅ RESTな設計

GET /users/123 POST /users DELETE /users/123
URLは名詞・操作はHTTPメソッド

⚠️ 「RESTはプロトコルである」は誤り。設計思想（アーキテクチャスタイル）

💡 かんたん解説

RESTは「設計のルール集」で、プロトコル（通信のやり方）ではない。URLは「モノの名前（名詞）」で、何をするか（動詞）はHTTPメソッドで決める、という約束事。

❌
非REST（悪い例）
/getUserById?id=1
/deleteUser
URLに動詞が混ざる
✅
REST（良い例）
GET /users/1
DELETE /users/1
URLは名詞、操作はメソッド

📋 2-2. RESTの4原則（試験の核心）

原則内容イメージ
① ステートレスサーバーはセッション状態を保持しない。各リクエストが自己完結毎回「初めまして」で会話
② 統一インターフェースGET/POST/PUT/DELETEのHTTPメソッドで操作を統一世界共通のボタン配置
③ リソース指向URLがリソース（データ）の場所を指す「棚番号」でモノの場所を特定
④ キャッシュ可能レスポンスをキャッシュして効率化できる一度調べた電話番号をメモ
💡 かんたん解説

RESTの4原則は「使いやすいAPIにするための4つのお約束」。特にステートレス（毎回はじめまして＝サーバーは前回を覚えていない）が一番出る。だから必要な情報は毎回まとめて送る。

🔘

操作はHTTPメソッド
📍
③リソース指向
URL＝モノの場所
💾
④キャッシュ可能
結果を再利用

⚖️ 2-3. HTTPメソッドと冪等性

冪等性（べきとうせい）＝「同じ操作を何回繰り返しても結果が同じ」

メソッド操作冪等性安全性使用例
GET取得○○商品一覧を見る
POST作成✗✗注文を確定する
PUT全更新○✗プロフィールを上書き
PATCH部分更新△✗メールアドレスだけ変更
DELETE削除○✗アカウントを削除

💡 POST（注文確定）を2回押すと2つ注文が入る → 非冪等 → 二重送信防止が重要

💡 GETとHEADのみが「安全なメソッド」（サーバーの状態を変えない）

💡 かんたん解説

「冪等（べきとう）」とは「何回やっても同じ結果」ということ。消しゴムで消した後にまた消しても同じ（DELETEは冪等）。でも注文ボタンを2回押したら2つ注文が入る（POSTは非冪等）。

安全 ✗

安全 ✗

△

冪等 ✓
安全 ✗

  • 4原則：ステートレス・統一インターフェース・リソース指向・キャッシュ可能

📊 3-1. JSON（現在の主流）

JSON（JavaScript Object Notation）の構造

{ "user": { "id": 123, "name": "山田太郎", "tags": ["manager", "tokyo"], ← 配列 [ ] "address": { ← ネスト（入れ子） "city": "東京" } } }

構造の特徴：{ }オブジェクト / [ ]配列 / "値"文字列 / 数値はそのまま / true/false / null

📄 3-2. XML（比較対象・レガシー）

<user> <id>123</id> <name>山田太郎</name> <tags> <tag>manager</tag> ← 配列は同名タグを繰り返す <tag>tokyo</tag> </tags> </user>

⚠️ 同じ情報でもXMLはJSONの約1.5〜2倍のデータ量になる

🔄 3-3. JSON vs XML 徹底比較

🎫 6-4. JWT（IDトークンの形式）

JWT（JSON Web Token）＝自己完結した証明書

// eyJhbGciOi... をBase64デコードすると { "sub": "user123", ← ユーザーID "email": "xxx@gmail.com", "exp": 1700000000 ← 有効期限 }

💡 JWTはサーバーに毎回問い合わせせず、トークン自体に情報が入っている（ステートレス）

💡 かんたん解説

JWTは「住民票」のようなもの。住民票には名前・住所・有効期限が印刷されているので、役所に確認しなくても内容を信頼できる。サーバー問い合わせ不要でステートレスを実現できる。

⚡ 7-1. RESTの課題

「ユーザーの名前と最新3件の注文だけ欲しい」場合

❌ RESTの問題

GET /users/123 → 全フィールド返ってくる (名前・住所・電話・生年月日…) GET /users/123/orders → 注文全件返ってくる (最新3件だけ欲しいのに)

① オーバーフェッチ（必要以上）
② アンダーフェッチ（2回必要）

VS

✅ GraphQL

query { user(id: 123) { name orders(last: 3) { id total } } }

1回で必要なデータだけ取得

📌 オーバーフェッチ＝必要以上のデータが返る / アンダーフェッチ＝必要なデータが足りず複数回リクエスト

💡 かんたん解説

GraphQLは「ファミレスのオーダー式ビュッフェ」。RESTは「セットメニュー」で要らないものも来るが、GraphQLは「欲しいものだけ」を一度に注文できる。エンドポイントは全メニュー共通で1つだけ。

😟 RESTの問題
・オーバーフェッチ：要らない情報も全部返る
・アンダーフェッチ：2回リクエスト必要
・エンドポイントが複数
😊 GraphQL解決

・1回のリクエストで完結
・エンドポイントは1つのみ

🔀 9-2. APIゲートウェイ

APIゲートウェイ＝複数のマイクロサービスへの「受付・警備・案内係」

APIゲートウェイの構造
外部クライアントスマホ・ブラウザ
↓
APIゲートウェイ認証確認・レート制限・ルーティング・ログ・集約
↓ ルーティング（振り分け）
注文サービス
決済サービス
在庫サービス

📌 セキュリティ・監視・制限を一カ所で管理。AWS API Gateway等がクラウドで提供

💡 かんたん解説

マイクロサービスは「専門店街」のしくみ。一つのお店（サービス）が閉まっても、他のお店は営業できる。APIゲートウェイは「総合受付」で、どの専門店に行けばいいか案内してくれる。

🏬 マイクロサービス構成
└ APIゲートウェイ（認証・ルーティング・制限を一括管理）
├ 注文サービス … 独立して動く
├ 決済サービス … 独立して動く
└ 在庫サービス … 独立して動く
